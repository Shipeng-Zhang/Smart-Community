# 项目重构说明：智慧社区居家安全与能耗系统

## 为什么重构

你提出的三个问题很关键：

1. 原场景实际价值不够贴近日常生活。  
2. 数据是提前生成，缺乏“实时变化”的动态感。  
3. 前后端耦合，不符合工程化系统交付习惯。  

本次重构将场景升级为 **“老旧社区居家安全与能耗协同管理”**，关注家家户户都会遇到的问题：厨房燃气泄漏、烟雾滞留、用电过载、漏水隐患、老人户低活动异常。

## 新系统定位

**挑战点**：多源传感信号融合 + 告警工单闭环效率（类似 OA 流程）

**系统价值**：

1. 对居民：降低燃气、电器、渗漏引发的安全风险。  
2. 对物业：把“告警 -> 派单 -> 处理 -> 闭环”做成在线流程。  
3. 对社区：形成持续运维数据，支撑节能和治理决策。

## 前后端分离架构

### 后端（API + 动态仿真）

位置：`/mnt/e/iot/src/smartcity_iot/`

1. `simulator.py`：实时生成住户多传感数据（非预生成）。  
2. `hub.py`：边缘规则识别、工单流转、通知中心、KPI 聚合。  
3. `server.py`：REST API（`/api/snapshot`、`/api/work-orders/action`、`/api/sim/control`）。

### 前端（独立静态应用）

位置：`/mnt/e/iot/frontend/`

1. `index.html`：OA 风格驾驶舱布局。  
2. `styles.css`：卡片化、流程看板化视觉语言。  
3. `app.js`：轮询拉取实时数据，动态渲染趋势、工单、告警、通知。

## 动态机制（非静态数据）

1. 后端后台线程持续推进仿真时间（每 tick 5 分钟）。  
2. 每个住户状态随时间段变化（做饭高峰、夜间、白天负载差异）。  
3. 随机/周期触发真实异常（燃气、烟雾、过载、漏水、门磁）。  
4. 工单自动升级与自动闭环机制，模拟真实 SLA 管理。  
5. 前端每 2.5 秒刷新，支持手动暂停、步进和工单动作按钮。

## 运行方式（本地 8080）

```bash
cd /mnt/e/iot
python3 start_community_system.py --port 8080 --households 24 --seed 42 --tick-seconds 1.2
```

浏览器访问：

`http://127.0.0.1:8080`
